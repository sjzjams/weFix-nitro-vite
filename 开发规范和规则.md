# 项目开发规范和规则

本文档旨在为 `weFix-nitro-vite` 项目提供一套统一的开发规范、技术约定和最佳实践，以确保代码质量、提升协作效率并降低维护成本。

---

## 1. 技术栈与架构

### 1.1. 核心框架
- **后端**: [Nitro](https://v3.nitro.build/) - 一个高性能的、与框架无关的服务器引擎。
- **前端**: [Vite](https://vite.dev/) + [React](https://react.dev/) - 使用 Vite 进行快速开发和构建，React (`.tsx`) 作为前端组件化方案。

### 1.2. 集成方式
- 通过 `nitro/vite` 插件将前后端无缝集成。
- **开发时**: Vite Dev Server 负责前端资源和 HMR，并将 API 请求代理到 Nitro Dev Server。
- **生产构建时**: 所有代码被打包成一个独立的 Nitro 服务器，实现单体部署。

### 1.3. 数据库
- **ORM**: 使用 **Prisma**。
- **Schema 定义**: `server/prisma/schema.prisma`。
- **重要规范**: 修改 `schema.prisma` 后，必须在 `package.json` 中确保 `prisma.schema` 路径配置正确，并运行 `npx prisma generate` 来更新 Prisma Client。

### 1.4. 状态管理
- 在原生 JS/TS 组件中，推荐使用自定义的 `EventEmitter` 模式进行简单的状态管理和组件通信，可参考 `app/database-explorer/state.ts` 的实现。

---

## 2. 目录结构与代码约定

- **`app/`**: 存放所有**前端**相关代码。
  - `app/components/`: 存放可复用的 React 组件。
  - `app/database-explorer/`: 数据库浏览器功能模块，采用原生 TS 和组件化思想。
  - `app/main.tsx`: 前端应用主入口，负责初始化和客户端路由。
- **`server/`**: 存放所有**后端** (Nitro) 代码。
  - `server/api/`: API 路由。每个文件或文件夹对应一个 API 端点。
  - `server/plugins/`: Nitro 插件，用于在服务器启动时注入服务或执行初始化逻辑（如数据库连接、日志服务）。
  - `server/prisma/`: 数据库 schema 定义文件。
- **`public/`**: 存放纯静态资源，如 `index.html` 中直接引用的 CSS、图片或无需构建的 JS 文件。

### 2.1. 代码风格
- **一致性**: 遵循项目现有代码风格。
- **格式化**: 推荐使用 Prettier 和 ESLint 进行代码格式化和检查。
- **注释**: 为新功能添加**函数级注释**和必要的文档，解释其功能、参数和返回值。

---

## 3. 环境与配置

### 3.1. 核心配置文件
- `vite.config.ts`: Vite 配置，主要用于集成 React 和 Nitro 插件。
- `nitro.config.ts`: Nitro 专属配置，**非常重要**。
  - `runtimeConfig`: 定义后端运行时配置（如数据库连接、Redis、JWT 密钥等）。这是配置敏感信息的**标准方式**。
  - `plugins`: 注册 Nitro 插件。**注意**: 插件路径必须是相对于**项目根目录**的完整路径 (e.g., `'./server/plugins/database'`)。
- `tsconfig.json`: TypeScript 配置。
  - **必须**包含 `"jsx": "react-jsx"` 以便 IDE 正确识别 React 17+ 的 JSX 语法。

### 3.2. 环境变量
- 所有敏感信息和环境相关配置都应通过 `.env` 文件定义。
- 在 `nitro.config.ts` 的 `runtimeConfig` 中通过 `process.env` 读取环境变量。
- **安全规范**: 前端代码**无法**也**不应该**直接访问后端的 `runtimeConfig`。

---

## 4. 开发流程与命令

- **安装依赖**: `npm install`
- **开发模式**: `npm run dev`
  - 访问 `http://localhost:3000` (或其他指定端口)。
- **构建生产包**: `npm run build`
  - 将整个应用打包到 `.output` 目录。
- **预览生产包**: `npm run preview`
  - 在本地运行生产版本的服务器。
- **数据库操作**:
  - 修改 `server/prisma/schema.prisma` 后，必须运行 `npx prisma generate`。

---

## 5. API 开发规范

- 在 `server/api/` 目录下创建 `.ts` 文件即可定义新的 API 端点。
- 文件名和目录结构决定了 API 的 URL 路径。例如 `server/api/user/profile.ts` 会创建 `/api/user/profile` 端点。
- 使用 `defineEventHandler` 来定义请求处理器。

---

## 6. 前端开发规范

### 6.1. 组件化
- **首选**: 使用 React 组件 (`.tsx`) 进行开发。
- **备选**: 对于简单的、独立的、或需要极致性能的场景，可以像 `database-explorer` 一样使用原生 TypeScript/JavaScript，并封装成模块化组件。

### 6.2. 路由
- 项目已实现一个简单的基于 `window.location.hash` 的客户端路由。
- **规范**: 动态内容应渲染在指定的“插座”元素（如 `#router-outlet`）中，避免意外影响页面上的静态元素及其样式。

### 6.3. 与后端交互
- 使用 `fetch` 或 `axios` 等库请求相对路径的 API (e.g., `/api/time`)。Vite 的代理会自动处理请求转发。

---

## 7. 测试规范

- **测试框架**: [Vitest](https://vitest.dev/)
- **测试环境**: 使用 `jsdom` 模拟浏览器环境，以便测试 DOM 操作。
- **配置文件**: `vitest.config.ts`
- **要求**: 鼓励为核心组件（尤其是原生 TS 实现的 UI 组件）编写单元测试，以验证其渲染和交互逻辑。
